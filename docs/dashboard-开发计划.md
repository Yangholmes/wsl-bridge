# Dashboard 开发计划（M4 前置优化）

## 1. 目标与依据

- 依据文档：`docs/wsl-bridge-uiux-design.md` 第 `5.1 概览（Dashboard）`。
- 目标：让用户在 10 秒内判断应用可用性、规则健康度和风险状态，并完成快捷跳转。

## 2. 功能范围

1. 应用状态卡片
- 状态枚举：`ready / running / stopped / error`。
- 数据来源：`get_runtime_status` 与拓扑最近扫描时间。

2. 规则状态卡片
- 指标：规则总数、已启用、运行中、错误数。
- 数据来源：`list_rules` + `get_runtime_status` 聚合。

3. 风险提示卡片
- 规则：当检测到 WSL `networking_mode = nat` 且无启用规则时提示风险。
- 数据来源：`scan_topology` + `list_rules`。

4. 快捷操作区
- `新建规则`：跳转 `/rules`。
- `扫描拓扑`：触发 `scan_topology` 刷新。
- `查看错误日志`：跳转 `/logs`。

5. 最近错误日志区
- 展示最近 24 小时错误日志（最多 8 条）。
- 数据来源：`query_logs({ level: "error", start_time, limit })`。

## 3. 交互与性能策略

1. 异步加载
- Dashboard 全部模块独立 Query 拉取。
- 路由懒加载，首屏显示骨架屏，避免切页白屏。

2. 反馈机制
- 刷新与扫描拓扑提供成功/失败提示。
- 卡片与日志区都支持 skeleton 占位。

3. 缓存策略
- 规则/运行态短周期刷新（5-10s）。
- 拓扑长缓存（60s）减少扫描频率。

## 4. 实施阶段

1. 阶段 A（已完成）
- 建立 Dashboard 正式页面与核心卡片。
- 接入规则/运行态/拓扑/错误日志数据。
- 完成快捷操作与基础提示。

2. 阶段 B（待 M4 前半）
- 增加顶部全局状态条（App/Engine/最近 Apply）。
- 增加“最近一次 Apply 成功/失败”摘要模块。

3. 阶段 C（待 M4 后半）
- 增加风险建议动作直达（例如一键打开规则向导）。
- 增加兼容性告警（权限不足、Hyper-V 不可用、WSL 未启动）。

## 5. 验收标准

1. Dashboard 打开后 1s 内出现骨架屏，数据到达后无布局抖动。
2. 四个核心模块（应用状态、规则状态、风险提示、快捷操作）可见且可用。
3. 在 NAT + 无规则场景下出现明确风险提示。
4. 最近错误日志可展示并可跳转到 Logs 页面进一步排查。
