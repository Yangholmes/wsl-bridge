### 二期规划与设计

完成二期需求讨论与方案设计，产出设计文档 `docs/phase2-traffic-monitor-design.md`：

**核心需求**

1. 流量监控：实时统计规则转发流量，Dashboard 曲线图展示
2. 请求日志：仿照 Nginx 设计 access log 和 error log 日志体系
3. 系统托盘：支持最小化到系统托盘，后台继续运行
4. 关闭拦截：拦截窗口关闭，提供关闭/最小化到托盘选项
5. MCP 扩展：暴露流量统计查询能力

**方案要点**

- 流量：Tauri Event Channel 秒级推送 + 内存滑动窗口 + SQLite 分钟聚合
- 图表：uPlot 库，实时刷新 + 固定窗口（30s/60s/120s 可选）
- 图表优化：Dashboard Tab 非活动时销毁图表，停止 Channel 监听，避免性能开销
- 日志：文件系统存储（access.log / error.log），按大小/时间轮转
- 日志轮转：手写 RollingFileWriter + tracing_appender::non_blocking，支持大小+时间双重轮转
- 数据库：traffic_stats 表使用 UUID 作为主键
- 界面：删除 Logs 页面，Dashboard 增加流量曲线区块
- 托盘：TrayIconBuilder + 菜单（显示窗口/退出应用），双击恢复窗口
- MCP：新增 `query_traffic_stats` / `get_traffic_window` 工具

**里程碑划分**（总工作量约 14 天）

| 里程碑 | 目标                  | 工时 | 主要交付物                                        |
| ------ | --------------------- | ---- | ------------------------------------------------- |
| M5.1   | 流量监控核心          | 4 天 | TrafficAggregator、Channel 推送、SQLite 持久化    |
