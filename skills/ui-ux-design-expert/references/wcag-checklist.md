# WCAG 检查清单（UI/UX）

## 使用说明

- 在进行可访问性评审、设计验收、组件规范编写时使用本清单。
- 优先覆盖高风险路径：注册、登录、搜索、支付、表单提交流程。
- 输出时按「问题 -> 影响 -> 修复建议 -> 验证方法」结构组织。

## 1. 感知性（Perceivable）

### 1.1 颜色与对比度

- 正文文本对比度至少满足 AA（4.5:1）。
- 大号文本（常见为 18pt 或 14pt 粗体及以上）满足 AA（3:1）。
- 关键图标、边框、状态标签与背景可辨识，不依赖微弱色差。
- 不仅用颜色表达状态，必须附加图标、文字或形状差异。

### 1.2 图像与媒体替代

- 具备语义作用的图片提供等效文本说明。
- 装饰性图片标记为装饰用途，避免干扰读屏。
- 视频提供字幕；有语音核心信息时提供文本稿。

### 1.3 内容结构

- 标题层级连续且语义正确（h1 -> h2 -> h3）。
- 表单项具备可见标签，提示信息与错误信息可感知。
- 表格用于数据展示时提供表头关联。

## 2. 可操作性（Operable）

### 2.1 键盘可用

- 所有核心功能可仅通过键盘完成。
- 焦点顺序符合视觉与任务逻辑。
- 焦点可见且高对比，不被样式移除。
- 提供“跳到主内容”等快速导航机制（复杂页面）。

### 2.2 交互容错

- 可点击区域足够大，避免误触（移动端尤其重要）。
- 不设置过短超时；有超时则允许延长或恢复。
- 动画可暂停、可关闭或尊重系统减少动效偏好。

### 2.3 导航一致性

- 全局导航位置与命名保持一致。
- 相同功能在不同页面采用一致交互方式。
- 链接与按钮语义清晰，不出现“点这里”类模糊文案。

## 3. 可理解性（Understandable）

### 3.1 文案与反馈

- 操作结果有即时反馈（成功、失败、加载、空状态）。
- 错误信息包含原因与可执行修复建议。
- 关键操作前后状态变化明确（例如“已保存”“保存失败”）。

### 3.2 表单体验

- 必填项、格式要求、输入示例明确展示。
- 校验时机合理（提交后或失焦后），避免频繁打断。
- 错误信息与字段建立关联，读屏可感知。

## 4. 健壮性（Robust）

### 4.1 语义与 ARIA

- 优先使用原生语义元素，避免无语义容器滥用。
- 仅在必要时补充 ARIA，避免与原生语义冲突。
- 自定义组件（如下拉、对话框、标签页）具备正确角色与状态属性。

### 4.2 兼容性验证

- 在主流浏览器与移动端环境验证关键流程。
- 至少进行一次读屏验证（如 NVDA/VoiceOver）。
- 在放大 200% 与缩放场景下检查布局与可读性。

## 输出模板

```markdown
## 可访问性审查结果

### 问题 1：登录按钮对比度不足
- 严重级别：高
- 影响：低视力用户难以识别主要操作
- 依据：WCAG 1.4.3（Contrast Minimum）
- 修复建议：将按钮背景调整为 #0B5FFF，文本维持 #FFFFFF
- 验证方式：重新计算对比度并进行键盘焦点可见性检查
```
